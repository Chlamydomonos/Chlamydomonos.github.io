什么是Git？

一. 版本控制

假设这样的场景——

  • 你正在开发一个项目。于是，你建立了一个源代码文件——a.cpp。

  • 第一天，你完成了项目一半的内容，并且把它们保存在了a.cpp里面。

  • 第二天，你又完成了项目四分之一的内容，并且删掉了第一天写的一小部分内容。可是，当你保存了a.cpp后，开始debug的时候，你才发现——你刚刚删掉的东西正是项目的关键代码。更可怕的是，你已经完全忘了你第一天是怎么写出的它们。

  • 第三天，你长了教训。你绞尽脑汁，修复了第一天和第二天写的代码，并且把这个文件另存为a_1.cpp。

  • 第四天，你完成了最后的四分之一，并把这个文件另存为a_2.cpp。

  • 第五天，你修复了一些bug，并把这个文件另存为a_3.cpp。

  • 第六天……

  • 第七天……

  • 最后，你的文件夹变成了这样——

    图片仅供参考

  • 于是，你不禁思考——有没有一个软件，自动帮我备份代码呢？

再假设这样的场景——

  • 你和你的同学一起开发一个项目。为此，你建立了一个源代码文件——a.cpp。
  • 第一天，你写了这个文件的一半，你的同学写了另一半，并把他的文件发给了你。
  • 现在，你有两个名为a.cpp的文件。你把你同学的内容复制粘贴进了你自己的a.cpp里，并把这个合并后的文件发给了你的同学。
  • 第二天，你们修复了程序中的若干bug。你修复完你的部分后，也收到了你的同学发给你的文件。
  • 你发现，你的文件在第1, 3, 15, 24, ……行有修改，你同学的文件在第4, 8, 19, 33, ……行有修改。
  • 于是，你只好又一行一行地检查代码，来看这行是否被你或者你的同学修改过。
  • 于是，你不禁思考——有没有一个软件，自动帮我合并不同版本呢？
这个软件，就是版本控制系统。

版本控制系统的功能——

  • 接收用户提交的代码，记录提交历史记录
  • 允许用户回退到自己之前的提交
  • 允许用户创建不同的版本并发布
  • 允许多个用户同时发布不同的版本，并合并这些版本成为新的版本

为了实现以上功能，版本控制系统会建立一个版本库。版本库可能在用户自己的电脑上，也可能位于网络中的服务器上。因此，版本控制系统分为两种类型——

  • 中心式——只有一个中心版本库，位于服务器上，所有用户需要通过网络向这个版本库提交版本。比较著名的有SVN等。
  • 分布式——每个用户的计算机上有一个版本库，用户直接向本地版本库提交版本，不同用户的版本库之间通过网络互相同步。在Git出现之前，比较著名的有BitKeeper等。

二. Git的诞生

在2005年之前，Linus创建的Linux社区一直使用BitKeeper作为版本控制系统。但是，2005年，BitKeeper决定对Linux社区收费。一怒之下，Linus用两个星期的时间，用C语言开发了一个简单但是极其强大的分布式版本控制系统，用于管理Linux的代码。这个版本控制系统也被开源，称为Git。

几年后，Git风靡世界，成为最著名的版本控制系统。

Git的原理

一个用Git管理的项目由三部分组成——工作区，暂存区和版本库。

工作区就是用户实际看到的文件夹。暂存区和版本库都位于项目文件夹中的.git文件夹中。用户可以使用add指令把文件加入暂存区，也可以使用checkout指令用暂存区的文件更新工作区。当用户使用commit指令时，暂存区中的文件受到的修改会被总结成一个提交，放入版本库。在版本库里，提交之间形成一张图，代表其互相的依赖关系。版本库中可以有若干称为分支的指针指向一些提交，代表一个版本。同时，版本库中还有一个特殊指针HEAD，指向新的提交将会被放到的位置。用户可以用reset指令把HEAD的内容覆盖到暂存区。此外，由于不同的版本库之间通过网络互相连接，用户也可以使用pull和push指令把自己的版本库与远程版本库同步。

如何使用Git

零. 打开Git Bash

Git Bash是一个用于在Windows系统上使用Git以及其他Linux命令的命令行工具。要想在某个文件夹中打开Git Bash，只需要右键单击该文件夹的空白部分，选择Git Bash Here。

在之后的教程中，凡是提到“在xxx文件夹中输入yyy”，均意为“在xxx文件夹打开的Git Bash中输入yyy”；凭空提到的“输入xxx”，均意为“在当前文件夹打开的Git Bash中输入xxx”。

一. 新建Git仓库

要想建立一个Git仓库其实是很简单的。你只需要输入git init，就可以把当前目录变成一个Git仓库。此时，该目录下应该会出现一个隐藏的.git文件夹。请勿删除该文件夹，原因见此处。

当然，也可以在想建立Git仓库的目录的上级目录中输入git init 目录名，来在该目录中建立Git仓库。

二. 克隆远程Git仓库

要想克隆一个远程的Git仓库，其实是更加简单的。你只需要输入git clone 远程仓库地址，Git就会在当前目录下创建一个与远程仓库同名的文件夹，并把远程仓库克隆到本地。

三. 添加文件到暂存区

众所周知，为了记录版本，我们需要把经过了修改的文件添加到暂存区。使用git add 文件名命令，即可把对应文件添加到暂存区。当然，你也可以输入git add 文件名1 文件名2 ……来一次把多个文件添加到暂存区。

此外，你也可以输入git add 文件夹名来把该文件夹内以及该文件夹的所有子文件夹内的所有文件添加到暂存区。

因此，由于.代表当前目录，输入git add .即可把整个项目的全部文件添加到暂存区。然而，这么做可能是错误的，因为像编译结果等文件并不应该被纳入版本库。

四. 把暂存区文件提交进版本库

这是Git最关键的操作之一。使用git commit -m "提交消息"命令，可以把暂存区提交进版本库。注意，这里的提交消息是必需的，最好在提交消息中简要描述你对文件做的更改。当然，提交之后，暂存区将被清空，在你下次提交更改之前，你需要重新进行添加文件到暂存区的工作。

五. 把本地仓库与远程仓库连接

对于从远程克隆下来的仓库，这步已经在克隆过程中完成了。因此，只有在本地创建的仓库需要执行此步骤。

输入git remote add 远程仓库名称 远程仓库地址来把本地仓库与远程仓库连接。注意，一般把第一个连接的远程仓库命名为origin。

六. 同步本地仓库与远程仓库

同步仓库的操作非常重要，但也是非常简单的。输入git pull 远程仓库名 分支名即可把远程仓库上某个分支的内容同步到本地，输入git push 远程仓库名 分支名即可把本地的内容同步到远程仓库。注意，在每次同步时，最好先pull再push。此外，如果远程仓库上还没有分支，你可以使用git push -u 远程仓库名 分支名来新建并同步分支。一般来说，当你第一次向远程仓库上传更改时，你需要使用git push origin master或git push origin main（如果你很重视政治正确）来在远程仓库中建立一个主分支。

总结

对于目前的班级活动工作，使用Git的工作流程基本为——

  • 克隆远程仓库到本地
  • 做出自己的修改
  • 使用git add .添加项目全部文件到暂存区（对于班级项目来说，这样并不会有危险，详见下一章教程）
  • 使用git commit -m "提交信息"提交更改
  • 使用git pull origin master及git push origin master同步更改到远程仓库
